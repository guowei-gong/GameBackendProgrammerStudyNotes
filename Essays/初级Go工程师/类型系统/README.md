## 类型推断
- 核心语法：变量 := 初始值
- 本质：一场“等价交换”。你给编译器一个具体的值，编译器还你一个确定的类型
- 关键点：
    - 非隐式转换：Go 不会在你看不见的地方偷偷改类型。int 就是 int，不会自动变 int32；
    - 默认偏好：整数默认为 int，浮点数默认为 float64；
    - 无类型常量：字面量，能根据上下文自动适配（只要不溢出）。

## 类型断言
- 场景：当面对一个 interface{} (或 any)，编译器不知道其真身时。
- 两种模式：
    - 激进派：v := x.(T)。猜错直接 panic；
    - 稳健派 (Comma-ok)：v, ok := x.(T)。猜错时 ok 为 false，v 为零值，程序不崩溃。
- Type Switch：
    ```go
    switch v := data.(type) {
    case string: // v 此时就是 string
    case int:    // v 此时就是 int
    }
    ```

## 小结
类型判断发生在编译时，类型断言发生在运行时。

## any 的代价
- 结构：双指针结构（类型指针 + 数据指针）。
- 副作用：
    - 内存逃逸：数据容易从栈跑到堆，增加 GC 压力；

        把一个具体的变量，比如 int，赋值给 any 时，由于 any 的底层结构是一个指针，Go 编译器往往无法在编译时确定这个变量的生命周期。

        直接访问：变量通常留在栈上，存取极快，函数结束自动回收。

        赋值给 any：变量往往会“逃逸”到堆上。堆分配需要申请内存，更麻烦的是，它会给 GC 带来巨大压力。

    - 性能损耗：间接寻址和运行时断言比直接访问慢。

## 泛型
- 类型安全：将错误从运行时提前到编译时；
- 性能优化：减少 any 带来的装箱和断言开销；
- 链路完整：输入和输出的类型**可以**保持一致（类型关联）。

## 总结
这个函数是否需要 ‘原样还回’ 输入的东西？

1. 是 $\rightarrow$ 用泛型（保持类型链路不断）。
2. 否，我只需要处理它 $\rightarrow$ 用接口/any。
3. 我不确定它是什么，但我得根据它的类型执行不同逻辑 $\rightarrow$ 用 any + Type Switch。

切记**不要过度使用泛型**。
